## Резюме (топ-5)

1. 🔴 **Бизнес-уравнение сравнивает несравнимое:** финансовые колонки (выручка/маржа/прибыль) суммируются только по прошедшим месяцам, а воронка (Запросы, Победы, Проекты) — по **всему периоду, включая будущие месяцы**. На годовом периоде «Запросы −56%» — это факт-на-сегодня против плана всего года. Реальное отставание к июлю ≈ −20%.
2. 🔴 **Тихие нули amoCRM:** при протухшем токене/недоступности amoCRM все счётчики воронки молча становятся нулями — без какой-либо индикации. Плюс фронт при 500 тихо падает на demo-данные (бейдж есть, но серый и незаметный). Дашборд может показывать неверные цифры, не сообщая об этом.
3. 🔴 **`/api/cashflow` делает до ~125 запросов к PlanFact за вызов** (баланс по каждому дню месяца + план по каждому дню на 3 мес вперёд), а семафора для PlanFact нет (у amoCRM — есть). Это главный источник «долго грузится» и каскадных 429 → 500 → demo-данные.
4. 🟡 **`data/notes.json` трекается в гите и одновременно пишется рантаймом на проде.** Первая же заметка на проде сделает рабочее дерево грязным → следующий `git pull` (деплой) законфликтует или затрёт заметки.
5. 🟡 **Часовые пояса:** границы месяцев и «текущий месяц» считаются в TZ сервера (UTC), а бизнес живёт по Москве. 1-го числа с 00:00 до 03:00 МСК «активный месяц» — прошлый; лиды, созданные в эти часы, попадают в соседний месяц.

## 1. Данные и формулы

### 1.1 🔴 NI-рассинхрон воронки и финансов в бизнес-уравнении

[business-equation-chart.tsx:171-196](../src/components/dashboard/business-equation-chart.tsx): цикл суммирует `requestsFact/requestsPlan/projectsPlan/winsFact` **до** проверки `if (!m.isPast) continue`, а финансовые поля — после. Итог на периоде Янв–Дек:

| Колонка | Факт | План | Показано | Честное сравнение к июлю |
|---|---|---|---|---|
| Запросы | 90 (к сегодня) | 203 (весь год) | **−56%** | план Янв–Июл = 112 → **−20%** |
| Победы | 19 | 61 = 203×30% | **−69%** | 112×30% ≈ 34 → **−44%** |
| Проекты по актам | 20 | 60 (весь год) | −67% | план Янв–Июл = 27 → −26% |

Финансовые колонки рядом (Выручка −28% и т.д.) считаются только по прошедшим месяцам — в одном графике два разных смысла отклонений. Именно из таких мест рождается Костино «местами что-то не сходится».

**Рекомендация:** для колонок воронки суммировать план так же, как финансы — только `isPast`-месяцы (одна строка: перенести суммирование планов под `continue`). Либо явно подписать «план на весь период» — но лучше первое: единая семантика «отклонение от плана на прошедшую часть периода».

То же в Култе: `CULT_BUDGET_PROJECTS = pastCount * 2` — уже по прошедшим ✓, а `totalRequestsPlan = 7×N месяцев периода` — по всем ✗ ([business-equation-chart.tsx:230](../src/components/dashboard/business-equation-chart.tsx), [route.ts:411-413](../src/app/api/kpi/route.ts)). Внутри одного контура — тоже вразнобой.

### 1.2 🔴 Тихие нули при недоступном amoCRM

[amocrm-client.ts:167-169, 272-274, 379-381, 431-433](../src/lib/amocrm-client.ts) — четыре guard'а вида `if (!BASE_URL || !ACCESS_TOKEN || !pipelineId) return []/0/{}`. Пустой env, протухший `AMOCRM_ACCESS_TOKEN`, опечатка в pipeline — и Запросы/Победы/Конверсия/Проекты = 0 **без единого сигнала** ни в логах, ни в UI. Ошибки HTTP из amoCRM бросаются ✓, но ловятся общим catch в route → 500 → фронт молча уходит на mock ([use-kpi.ts:70-79](../src/lib/use-kpi.ts)).

**Рекомендация (fail loud):**
- guard'ы заменить на `throw new Error("amoCRM не сконфигурирован: ...")` — либо, если частичная деградация желанна, добавить в `KpiResponse` поле `sources: { amocrm: "ok"|"error", planfact: "ok"|"error" }` и показывать бейдж «⚠️ amoCRM недоступен — воронка не загружена».
- Demo-фолбэк на фронте либо убрать (прод не должен показывать вымышленные цифры), либо сделать бейдж кричащим (красный, с причиной).

### 1.3 🟡 Часовые пояса (UTC vs МСК)

- [route.ts:86-87](../src/app/api/kpi/route.ts): `currentMonth` из `new Date()` в TZ сервера (UTC). 1-го числа 00:00–02:59 МСК прод считает текущим прошлый месяц: «Цели месяца» показывают не тот месяц, `isPast` смещён.
- [amocrm-client.ts:173-174](../src/lib/amocrm-client.ts): `new Date(endDate + "T23:59:59")` парсится как UTC → границы месяца для amoCRM сдвинуты на 3 часа относительно МСК; лиды первых трёх часов месяца бакетятся в предыдущий.
- [cashflow route:15-16](../src/app/api/cashflow/route.ts): `new Date()` то же самое — «сегодня» до 03:00 МСК — вчера.

**Рекомендация:** зафиксировать бизнес-TZ константой (`Europe/Moscow`) и считать `currentMonth`/границы через неё (например, `toLocaleString("sv", { timeZone })`). Проверить TZ VPS (`timedatectl`).

## 2. Архитектура

### 2.1 🔴 Шторм запросов и отсутствие семафора PlanFact

Холодная загрузка дашборда (один пользователь, один контур) порождает порядка **150+ внешних запросов**:

| Эндпоинт | PlanFact | amoCRM |
|---|---|---|
| `/api/kpi` (Цели месяца, 1 мес) | ~8 | ~12 |
| `/api/kpi` (глобальный период, 12 мес) | ~19 | ~35 |
| `/api/cashflow` | **до ~125** (баланс/день + cashflow/день×3 мес, [route:33-60](../src/app/api/cashflow/route.ts)) | — |

- У amoCRM есть семафор (4) + inflight-дедуп ([amocrm-client.ts:75-134](../src/lib/amocrm-client.ts)) — грамотно. У **PlanFact — ничего**: `Promise.all` лупит всё разом → 429 (наблюдали живьём) → 500 → demo-данные.
- Кнопка «Обновить» сбрасывает **оба тега целиком** ([api/refresh](../src/app/api/refresh/route.ts)) → повторный полный шторм.
- Next data cache (15 мин/24 ч, [planfact-client.ts:3-19](../src/lib/planfact-client.ts)) смягчает тёплые загрузки, но после деплоя (`npm run build` пересоздаёт `.next`) кеш пуст — первый пользователь ловит всё.

**Рекомендации (по возрастанию усилий):**
1. Семафор+дедуп для `pfFetch` по образцу amoCRM (~30 строк, копия паттерна) — уберёт 429.
2. `/api/cashflow`: прошлые дни неизменны — кешировать помесячно/навсегда (или брать баланс раз в неделю + считать дельты); будущее — **одним** запросом `getCashFlow(start, end, standardPeriod: "Day")` вместо 92 однодневных (PlanFact отдаёт `totalValuesByPeriod` — уже поддержано типами [planfact-client.ts:80-99](../src/lib/planfact-client.ts)). Потенциал: ~125 → **3–5 запросов**.
3. Снапшот-слой (раздел 4) — закрывает UX полностью.

## 4. «Снимок» данных — рекомендация

**Запрос:** мгновенная загрузка из снапшота всегда, актуализация — фоном/по кнопке; без новой инфраструктуры.

**Хорошая новость:** Next data cache уже работает как stale-while-revalidate (после TTL отдаёт старое и обновляет фоном). Проблема в двух местах: (а) после каждого деплоя кеш пуст (build пересоздаёт `.next`) — «холодный шторм»; (б) кеш неявный, нет «времени снимка» для UI. Поэтому:

### Рекомендуемая архитектура: файловый снапшот + двухфазный фетч (вариант B) — «то же, что notes.json, только для KPI»

**Сервер** (~40 строк):
- После успешного расчёта `/api/kpi` — атомарная запись `data/snapshots/kpi-{entity}-{start}-{end}.json` с `{ snapshotAt, payload }` (write tmp → rename). Каталог `data/` уже существует и обслуживается этим же паттерном (notes.json).
- `GET /api/kpi?snapshot=1` — читает файл и отвечает мгновенно (без единого внешнего запроса); 404, если снапшота ещё нет. Аналогично для `/api/cashflow`.
- Снапшоты пишутся только для фактических ключей запросов — а фронт по умолчанию запрашивает 3 канонических (текущий месяц, глобальный период, полный год) × 2 контура ≈ 6–8 файлов. Не «1000 баз»: плоские JSON-файлы, ноль новых сервисов.

**Фронт** (`use-kpi`, ~20 строк):
1. Сразу `fetch(?snapshot=1)` → мгновенный рендер + подпись «данные от 14:23» (из `snapshotAt` — заодно честное серверное время для «Обновлено», UI-п.4/9).
2. Параллельно обычный `fetch` → по приходу данные подменяются, подпись обновляется.
3. Live упал (429/500) → **остаёмся на снапшоте** с плашкой «показан снимок от 14:23, обновление недоступно» — demo-фолбэк с вымышленными данными больше не нужен.

**Опционально (+15 мин):** прогрев — crontab на сервере раз в 15 мин дёргает 6 канонических URL. Снапшоты и Next-кеш всегда тёплые, включая сразу после деплоя; пользователи вообще никогда не ждут PlanFact.

**Почему не альтернативы:**
- *localStorage-кеш на клиенте* — быстро, но снимок per-браузер (первый вход всё равно ждёт), нет серверного «времени данных». Можно добавить позже как вишенку.
- *SQLite/Redis/отдельная БД* — то самое «ещё один слой, который может ломаться»; для 8 JSON-ключей не оправдано.
- *Только тюнинг Next-кеша* — не переживает деплой и не даёт UI «снимок от HH:MM».

**Связка с разделом 2.1:** снапшот закрывает UX, но семафор PlanFact (п.1) и оптимизацию cashflow (п.2) делать всё равно — иначе фоновая актуализация будет продолжать ловить 429.
